# docs/plans

**進行中** の実装計画を置く場所。大きめのタスクに着手する前に、エージェント（あるいは人間）がここにプランを書いてから実装に入る。

## いつ書くか

- 複数ファイル・複数パッケージをまたぐ実装
- アーキテクチャに影響する追加（新しい public API、新しい依存方向）
- 不可逆な変更（DB マイグレーション、外部サービス連携の追加）
- 方針を複数検討して1つを選ぶ必要があるとき

## ファイル名

`NNNN-短い題名.md` の連番形式（例: `0001-meeting-search-rewrite.md`）。議論の足跡を残すため、番号は既存の最大値+1。

## テンプレ

```markdown
# <タイトル>

## ゴール
一文で。何が達成できたら完了か。

## 背景 / 動機
なぜ今これをやるのか。読者が前提知識ゼロでも分かるように。

## スコープ
含むもの・含まないもの。後者のほうが重要。

## アプローチ
手順を順番に。検討した代替案があれば「なぜ選ばなかったか」を書く。

## 影響範囲
触るパッケージ・ファイル。関連する AGENTS.md・.claude/rules。

## 検証
どうやって動作確認するか。`bun run agent:check` 以外に必要なら明記。

## オーナー / 期限
担当（人間 or エージェント）と目標日。
```

## プラン完了後

- 設計ドキュメントとして残したい → `docs/specs/` に昇格
- 判断の記録として残したい → `docs/decisions/` に ADR を書く
- どちらでもない → 削除して git ログに任せる
