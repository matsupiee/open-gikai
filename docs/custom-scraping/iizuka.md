# 飯塚市議会 カスタムスクレイピング方針

## 概要

- サイト: https://www.city.iizuka.lg.jp/site/shigikai/
- 分類: 独自 CMS による HTML ページ公開（既存アダプターでは対応不可）
- 文字コード: UTF-8
- 特記: 会議録は PDF 形式で提供。録画中継は YouTube 外部リンク（`youtu.be`）として提供。独立した全文検索システムは存在しない。

---

## URL 構造

| ページ | URL パターン |
| --- | --- |
| 年度別インデックス一覧 | `https://www.city.iizuka.lg.jp/site/shigikai/list18.html` |
| 年度別会議一覧 | `https://www.city.iizuka.lg.jp/site/shigikai/list18-{年度ID}.html` |
| 会議詳細ページ | `https://www.city.iizuka.lg.jp/site/shigikai/{記事ID}.html` |
| PDF 本体 | `https://www.city.iizuka.lg.jp/uploaded/attachment/{添付ID}.pdf` |
| YouTube 録画 | `https://youtu.be/{動画ID}` |

## 会議種別

各年度ページには以下の種別が含まれる：

- 本会議（定例会・臨時会）
- 常任委員会（総務、福祉文教、経済建設、協働環境）
- 議会運営委員会
- 特別委員会（予算特別、決算特別 等）

---

## 各会議詳細ページの構成

会議詳細ページ（`/site/shigikai/{記事ID}.html`）には以下が含まれる：

- 会議名・開催日程の一覧（テキスト形式）
- PDF リンク: 会期日程、議案付託一覧、目次、各開催日の会議録
- YouTube リンク: 各開催日・議員質問ごとの録画中継

### PDF リンク構造

```
/uploaded/attachment/{添付ID}.pdf
```

同一会議の複数 PDF は連番に近い添付 ID が付与される（例: 8167, 8169, 8171, ...）。

### YouTube リンク構造

```
https://youtu.be/{11文字の動画ID}
```

一般質問は議員ごとに動画が分割されている。議案質疑も複数パートに分割される場合がある。

## スクレイピング戦略

### Step 1: 年度別会議一覧の収集

`list18.html` から各年度ページの URL（`list18-{年度ID}.html`）を抽出する。

### Step 2: 会議詳細ページ ID の収集

各年度ページ（`list18-{年度ID}.html`）から会議詳細ページへのリンク（`/site/shigikai/{記事ID}.html`）を抽出する。

### Step 3: PDF リンクの収集

各会議詳細ページから `/uploaded/attachment/{添付ID}.pdf` 形式の PDF リンクを抽出する。

- 会期日程・議案付託一覧・目次・各日の会議録が別ファイルで提供される
- 「作成次第、掲載いたします」と記載されている場合は会議録未公開のため、一定期間後に再クロールが必要

### Step 4: PDF の取得・解析

各会議録 PDF を取得してテキストを抽出する。

- 発言者・発言内容の構造は PDF 内のテキストに依存するため、PDF パーサーによる解析が必要
- 開催日は詳細ページの HTML から取得可能

---

## 注意事項

- 会議録 PDF の公開は会議終了後に遅延する場合がある（「作成次第、掲載いたします」の記載あり）
- YouTube 動画は「掲載した日から起算して7年が経過した年度末まで」公開。会議録本体（PDF）とは別管理
- 会議録テキストは PDF 形式のため、HTML での全文スクレイピングは不可
- 年度 ID は連番ではなく、年度追加時に新しい ID が付与されるため、`list18.html` から動的に取得する必要がある

---

## 推奨アプローチ

1. **インデックスから動的取得**: `list18.html` の年度リンクを起点に会議 ID を収集し、ハードコードを避ける
2. **PDF テキスト抽出**: pdfjs や pdfminer 相当のライブラリで PDF を解析する
3. **未公開ページの検出**: 「作成次第、掲載いたします」の文字列を検出して未公開ページをスキップする
4. **レート制限**: 自治体サイトのため、リクエスト間に 1〜2 秒の待機時間を設ける
