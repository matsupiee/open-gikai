# 全自治体議会データ収集

---

## 1. プロジェクト概要

日本全国約 1,800 自治体の議会データ（議事録テキスト・日程・開催情報）を自動収集し、統合検索サービスを構築することを目的とする。

**収集対象データ**

- 議事録テキスト — 本会議・委員会の発言録（発言者・内容・日付）
- 日程・開催情報 — 定例会・臨時会の日時・会議種別
- 議員情報 — 氏名・会派・期数など（将来対応）

---

## 2. 現状分析：自治体議会システムの生態系

最大の課題は、全国自治体が複数の異なるシステム・形式でデータを公開していることである。
スクレイピングはシステムごとに専用アダプターを作る「マルチアダプター方式」を採用する。

### 2.1 主要会議録システムの分布

| 会議録システム     | ベンダー             | 採用割合（推定） | 難易度            | 実装状況  |
| ------------------ | -------------------- | ---------------- | ----------------- | --------- |
| **DiscussNet SSP** | 会議録研究所         | ★★★★☆            | 中（JSON API）    | ✅ 実装済 |
| **DB-Search**      | 大和速記情報センター | ★★★☆☆            | 中（HTML + CSRF） | ✅ 実装済 |
| **検索システム**   | 複数ベンダー         | ★★★☆☆            | 中（Shift-JIS）   | ✅ 実装済 |
| **議事録.com**     | 議事録.com           | ★★☆☆☆            | 中（voiweb.exe）  | ✅ 実装済 |
| Sophia             | 神戸綜合速記         | ★★☆☆☆            | 中                | 未対応    |
| VOICES             | フューチャーイン     | ★★☆☆☆            | 中                | 未対応    |
| 独自 HTML          | 各自治体             | ★★★☆☆            | 高                | 未対応    |
| PDF ファイル       | （システムなし）     | ★★☆☆☆            | 高                | 未対応    |

### 2.2 公開形式の 3 分類

**① 会議録検索システム経由**

- HTML 構造・API がベンダーごとに規則的 → アダプター 1 本で複数自治体をカバー可能
- DiscussNet SSP は `ssp.kaigiroku.net` への JSON API で取得可能
- DB-Search は CSRF トークン・セッション管理が必要な HTML スクレイピング
- 検索システム（kensakusystem.jp）は Shift-JIS エンコーディングへの対応が必要

**② 独自 CMS による HTML 公開**

- HTML 構造がバラバラ → 個別対応が必要（優先度は下げて対応）

**③ PDF ファイル直接公開**

- PDF → テキスト変換が必要
- 将来フェーズで対応予定

### 3.1 技術スタック

| ジョブキュー   | Cloudflare Queue                          |
| スケジューラー | Cloudflare Cron（1 分ごと）               |
| API            | oRPC                                      |
| フロントエンド | TanStack Start + React                    |
| 認証           | Better Auth                               |
| インフラ       | Cloudflare Workers（wrangler deploy）     |

### 3.2 スクレイピング実行方式

データ収集は **2 つの実行方式** を使い分ける。

| 方式 | 対象期間 | 用途 |
| --- | --- | --- |
| **local-bulk-scraper**（`apps/local-bulk-scraper`） | 2025 年以前の過去データ | ローカルで一括実行し NDJSON を出力 → DB にインポート |
| **scraper-worker**（`apps/scraper-worker`） | 2026 年以降の最新データ | Cloudflare Queue 経由でリアルタイムに逐次取得 |

#### local-bulk-scraper（過去データ一括取得）

```
CLI 実行（scrape:ndjson）
  ↓
全 enabled 自治体をホストごとにグルーピングし並列スクレイピング
  ↓
meetings.ndjson / statements.ndjson / statement_chunks.ndjson を出力
  ↓
import:ndjson で DB に一括インポート
```

#### scraper-worker（最新データ逐次取得）

```
管理画面でジョブ作成 → scraper_jobs に INSERT (status: pending)
  ↓
Cron（1分ごと）が pending → queued に更新し dispatchJob で最初のメッセージをキュー投入
  ↓
Queue コンシューマーが各メッセージを handleQueueMessage() でルーティング
  ↓
各アダプターのハンドラーが 1 ページ単位で取得 → 次ページのメッセージを再キュー
  ↓
最終ハンドラーが meetings テーブルに保存（externalId で重複スキップ）
  ↓
process-meetings が meetings の rawText → statements に分割
  ↓
build-chunks が statements をスピーカーごとにチャンク化 → embedding を生成して statement_chunks に保存
```

### 3.3 メッセージフロー

| アダプター     | 第 1 メッセージ           | 第 2 メッセージ         |
| -------------- | ------------------------- | ----------------------- |
| discussnet_ssp | `discussnet-ssp:schedule` | `discussnet-ssp:minute` |
| dbsearch       | `dbsearch:list`           | `dbsearch:detail`       |
| kensakusystem  | `kensakusystem:list`      | `kensakusystem:detail`  |
| gijiroku_com   | `gijiroku-com:list`       | `gijiroku-com:detail`   |

## 4. データモデル設計

### 4.1 DB テーブル構成

| テーブル名                              | 主なカラム                                                                                       | 説明                                           |
| --------------------------------------- | ------------------------------------------------------------------------------------------------ | ---------------------------------------------- |
| `municipalities`                        | id, code, name, prefecture, system_type_id, base_url, enabled, **population, population_year**   | 自治体マスタ（約 1,800 件）                    |
| `system_types`                          | id, name, description                                                                            | 会議録システム種別マスタ（4 種）               |
| `meetings`                              | id, municipality_id, title, meeting_type, held_on, source_url, external_id, raw_text, status     | 会議単位（status: pending / done）             |
| `statements`                            | id, meeting_id, kind, speaker_name, speaker_role, content, content_hash, content_tsv             | 発言単位（全文検索対象）                       |
| `statement_chunks`                      | id, meeting_id, speaker_name, chunk_index, content, statement_ids, content_hash, **embedding**   | 発言チャンク（意味検索対象、embedding を保持） |
| `scraper_jobs`                          | id, municipality_id, status, year, processed_items, total_inserted, total_skipped, error_message | ジョブ管理                                     |
| `scraper_job_logs`                      | id, job_id, level, message, created_at                                                           | ジョブログ                                     |
| `policy_tags` / `statement_policy_tags` | 政策タグ（将来利用）                                                                             |                                                |

**enums**:

- `scraper_job_status`: `pending` / `queued` / `running` / `completed` / `failed` / `cancelled`
- `log_level`: `info` / `warn` / `error`

### 4.2 municipalities テーブルの重要カラム

| カラム            | 型        | 説明                                                            |
| ----------------- | --------- | --------------------------------------------------------------- |
| `code`            | text      | 総務省 地方公共団体コード（6 桁、ユニーク）                     |
| `system_type_id`  | text (FK) | system_types.id を参照。シードスクリプトが baseUrl から自動判定 |
| `base_url`        | text      | 議事録トップ URL（スクレイピング起点）                          |
| `enabled`         | boolean   | false = robots.txt / 利用規約違反で除外                         |
| `population`      | integer   | 住民基本台帳に基づく人口（令和 7 年 1 月 1 日現在）             |
| `population_year` | integer   | 人口データの基準年（現在は 2025）                               |

### 4.3 全文検索・意味検索

- **全文検索**: `statements.content_tsv`（GENERATED ALWAYS、tsvector）+ `to_tsquery`
- **意味検索**: `statement_chunks.embedding`（vector(1536)、pgvector）+ コサイン類似度

```sql
-- 全文検索
WHERE statements.content_tsv @@ to_tsquery('simple', '検索語')

-- 意味検索（コサイン類似度）
ORDER BY statement_chunks.embedding <=> '[...]'::vector
```

### 4.4 statement_chunks の設計思想

- `statements` はテキスト分割の最小単位（1 発言 = 1 statement）
- 意味検索のためには発言者ごとにまとめた「チャンク」が適切なため、`statement_chunks` に集約
- embedding は `statements` ではなく `statement_chunks` にのみ保持（`statements.embedding` は 0003 マイグレーションで削除済み）
- 重複する手続き的発言（「賛成」「異議なし」等）は deduplicate してチャンク数を削減

## 5. アダプター設計

### 5.1 DiscussNet SSP アダプター（実装済）

**対象 URL**: `https://ssp.kaigiroku.net/tenant/{slug}/MinuteSearch.html`

1. `tenant.js` から `tenantId` を取得
2. API エンドポイント（`/MinuteSearch.html` 同一ホスト）に POST → スケジュール一覧取得
3. 各 `schedule_id` に対して議事録詳細を POST → meetings 保存

### 5.2 DB-Search アダプター（実装済）

**対象 URL**: `dbsr.jp` を含む URL

1. トップページ GET → CSRF トークン・Cookie・検索エンドポイント ID を取得
2. 年度フィルターで POST → 検索結果（1 ページ目）+ セッション ID を取得
3. 全ページをページングして文書 ID と URL を収集
4. 各詳細 URL を取得 → meetings 保存

### 5.3 検索システム（kensakusystem）アダプター（実装済）

**対象 URL**: `kensakusystem.jp` を含む URL

- URL パターンが 3 種（Sapphire / CGI / index.html）あり、それぞれに対応
- Shift-JIS エンコーディングのレスポンスをバイトレベルで解析
- 全角数字 → 半角変換・日本語元号（令和・平成・昭和）→ 西暦変換を実装

### 5.4 議事録.com（gijiroku_com）アダプター（実装済）

**対象 URL**: `gijiroku.com` を含む URL（voiweb.exe ベースの CGI システム）

1. `gijiroku-com:list` — ベース URL + 年度を指定して会議一覧を取得。各会議の `fino`（ファイル番号）・`kgno`（会議番号）・`unid`（一意 ID）を収集
2. 各会議に対して `gijiroku-com:detail` をキューに投入
3. 詳細ハンドラーが会議本文を取得 → meetings 保存 → statements 分割

### 5.5 将来対応：Sophia / VOICES アダプター

- `dispatch-job.ts` でルーティング先を追加するだけで対応可能

### 5.6 将来対応：PDF アダプター

- 詳細は `docs/PDF議事録スクレイピング戦略.md` を参照

## 6. シードデータ

### 6.1 自治体マスタ

- **ファイル**: `packages/db/src/seeds/municipalities.csv`（1,794 自治体）
- **カラム**: 団体コード, 都道府県名, 市区町村名, カナ, カナ, 議事録検索 URL, 人口, 人口基準年
- **人口データ**: 総務省「住民基本台帳に基づく人口・世帯数（令和 7 年 1 月 1 日現在）」から全件取得済み

### 6.2 system_types マスタ

スキーマで以下の 4 種を定義している：

| name             | 説明                                 |
| ---------------- | ------------------------------------ |
| `discussnet_ssp` | DiscussNet SaaS（ssp.kaigiroku.net） |
| `dbsearch`       | DB-Search（dbsr.jp）                 |
| `kensakusystem`  | 検索システム（kensakusystem.jp）     |
| `gijiroku_com`   | 議事録.com（gijiroku.com）           |

### 6.3 systemType 自動判定ロジック

`seed-municipality.ts` が `baseUrl` のパターンから `systemType` を自動判定して登録する：

```
ssp.kaigiroku.net  → discussnet_ssp
dbsr.jp            → dbsearch
kensakusystem.jp   → kensakusystem
gijiroku.com       → gijiroku_com
```

## 7. API 設計

### 7.1 ルーター構成（packages/api）

```
appRouter
├── statements
│   ├── search            # 全文検索
│   ├── semanticSearch    # 意味検索（pgvector）
│   └── ask               # AI 回答生成
├── meetings
│   ├── list              # 会議一覧（フィルター・ページング）
│   └── statements        # 会議の発言一覧
├── municipalities
│   └── list              # 自治体一覧（キーワード・人口順ソート対応）
└── scrapers              # 要認証（Better Auth）
    ├── listJobs          # ジョブ一覧
    ├── createJob         # ジョブ作成
    ├── createBulkJobs    # 全 enabled 自治体のジョブ一括作成
    ├── getJob            # ジョブ詳細
    ├── getJobLogs        # ジョブログ取得
    ├── cancelJob         # ジョブキャンセル
    ├── deletePendingJobs # pending ジョブ一括削除
    ├── listMunicipalities# スクレイピング対象自治体一覧
    ├── reprocessStatements # 発言の再チャンク・再埋め込み
    ├── progressByPrefecture  # 都道府県別進捗
    ├── progressByMunicipality # 自治体別進捗
    └── progressByYear    # 年度別進捗
```

### 7.2 自治体一覧 API（municipalities.list）

- `query`: キーワード検索（name / prefecture / baseUrl の部分一致）
- `sortBy`: `"code"`（都道府県・団体コード順）/ `"population"`（人口降順）
- `offset`, `limit`: オフセットページング
- レスポンス: `municipalities[]`（population 含む）+ `total` 件数

## 9. 主な課題と対策

### 9.1 アクセス制限・ブロック対策

- **対策 ①** リクエスト間隔：各ハンドラーで `delay.ts` により適切なウェイトを設定
- **対策 ②** Cloudflare Queue の `maxConcurrency` で同時実行数を制限

### 9.2 robots.txt・利用規約

- `municipalities.enabled = false` で対象外とする
- 収集前に robots.txt を手動確認してフラグを設定

### 9.3 Shift-JIS 対応

- kensakusystem のレスポンスは Shift-JIS エンコーディング
- `TextDecoder("shift-jis")` + バイトレベルの 2 バイト文字判定で対応（`kensakusystem/_shared.ts`）

### 9.4 CSRF / セッション管理（DB-Search）

- トップページからトークンと Cookie を取得し、後続リクエストに引き継ぐ
- セッション ID をページング時にリクエストヘッダーに付与

## 11. 法的・倫理的留意事項

> ⚠️ **必ず守ること**
>
> - `robots.txt` を確認し、Disallow されているパスはクロールしない
> - 利用規約でスクレイピング禁止の記載がある場合は `enabled = false` として対象外とする
> - データは公開情報のみ収集（ログイン不要のページのみ）
> - サーバーに過大な負荷をかけない（クロール間隔・並列数を制限）
> - データの帰属を明示し、出典として各自治体の URL をリンク表示する
