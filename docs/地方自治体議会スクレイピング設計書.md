# 全自治体議会データ収集プロジェクト
**スクレイピング設計書 v1.0　　2026年3月**

---

## 1. プロジェクト概要

日本全国約1,700自治体の議会データ（議事録テキスト・日程・開催情報・議員情報）を自動収集し、統合検索サービスを構築することを目的とする。

**収集対象データ（3種）**

- 議事録テキスト — 本会議・委員会の発言録（発言者・内容・日付）
- 日程・開催情報 — 定例会・臨時会の日時・会議種別
- 議員情報 — 氏名・会派・期数など

---

## 2. 現状分析：自治体議会システムの生態系

最大の課題は、1,700自治体が複数の異なるシステム・形式でデータを公開していることである。スクレイピングはシステムごとに専用アダプターを作る「マルチアダプター方式」が必要になる。

### 2.1 主要会議録システムの分布

| 会議録システム | ベンダー | 採用割合（推定） | 難易度 |
|---|---|---|---|
| DiscussNet | NTT-AT / 会議録研究所 | ★★★★★（最多） | 中（HTML規則的） |
| DB-Search | 大和速記情報センター | ★★★☆☆ | 中 |
| Sophia | 神戸綜合速記 | ★★☆☆☆ | 中 |
| VOICES | フューチャーイン | ★★☆☆☆ | 中 |
| 独自システム | 各自治体 | ★★★☆☆ | 高（個別対応） |
| PDF/HTMLファイル | （システムなし） | ★★☆☆☆ | 高（パーサー必要） |

### 2.2 公開形式の3分類

**① 会議録検索システム経由（DiscussNet 等）**
- HTML構造が各ベンダーで規則的 → アダプター1本で数百自治体をカバー可能
- DiscussNetは全国最多。URLパターンも統一されており最も取り組みやすい

**② 独自CMSによるHTML公開**
- 自治体が独自開発したウェブサイトに直接掲載
- HTMLの構造がバラバラ → 個別対応が必要（優先度は下げて対応）

**③ PDFファイル直接公開**
- PDF→テキスト変換（pdfplumber / pdfminer）が必要
- スキャンPDFはOCRが必要でさらに難易度が上がる

## 4. アダプター設計

### 4.1 DiscussNet アダプター（最優先）

NTT-ATのDiscussNetは全国最多導入システム。URLパターンが規則的で、1本のアダプターで数百自治体をカバーできる。

```python
class DiscussNetAdapter:
    BASE = 'https://{domain}/discuss/'

    def fetch_sessions(self, base_url) -> list[Session]:
        # 定例会・臨時会の一覧ページをスクレイピング
        soup = self._get_soup(base_url + 'index.htm')
        return [self._parse_session(row) for row in soup.select('table.kaigiroku tr')]

    def fetch_minutes(self, session_url) -> list[Speech]:
        # 発言者・発言内容を1発言単位で取得
        soup = self._get_soup(session_url)
        return [self._parse_speech(block) for block in soup.select('div.hatsugen')]

    def fetch_members(self, base_url) -> list[Member]:
        # 議員名簿ページから氏名・会派・期数を取得
        soup = self._get_soup(base_url + 'members.htm')
        return [self._parse_member(row) for row in soup.select('table.giin tr')]
```

### 4.2 PDF アダプター

PDFで公開している自治体向け。pdfplumberでテキスト抽出後、正規表現で発言者・発言内容を分離する。

```python
import pdfplumber, re

class PDFAdapter:
    SPEAKER_PATTERN = re.compile(
        r'([^\n]+?(?:議員|市長|町長|村長|議長|委員長))[　 ]{1,3}'
    )

    def parse(self, pdf_path) -> list[Speech]:
        with pdfplumber.open(pdf_path) as pdf:
            text = ''.join(pg.extract_text() or '' for pg in pdf.pages)
        return self._split_by_speaker(text)
```

## 5. データモデル設計

### 5.1 DBテーブル構成

| テーブル名 | 主なカラム | 説明 |
|---|---|---|
| `municipalities` | id, name, prefecture, url, system_type | 自治体マスタ（約1,700件） |
| `sessions` | id, municipality_id, name, date, session_type | 会議セッション（定例会・臨時会） |
| `minutes` | id, session_id, speaker, speech, order_no | 議事録発言単位（全文検索対象） |
| `members` | id, municipality_id, name, party, term_count | 議員情報 |
| `scrape_logs` | id, municipality_id, status, scraped_at, error | クロール実行ログ |

### 5.2 全文検索設計

PostgreSQL + PGroonga（日本語対応全文検索拡張）を採用。Elasticsearchより運用が軽量で、小〜中規模には十分。

```sql
-- PGroongaによる日本語全文検索インデックス
CREATE EXTENSION pgroonga;
CREATE INDEX idx_minutes_speech ON minutes
  USING pgroonga (speech pgroonga_text_full_text_search_ops_v2);

-- 検索クエリ例
SELECT m.speaker, m.speech, s.name, s.date, mu.name AS city
FROM minutes m
JOIN sessions s ON m.session_id = s.id
JOIN municipalities mu ON s.municipality_id = mu.id
WHERE m.speech &@~ '少子化 対策'  -- 日本語AND検索
ORDER BY s.date DESC LIMIT 20;
```

## 6. 主な課題と対策

### 6.1 アクセス制限・ブロック対策

- **対策①** リクエスト間隔：各自治体サーバーに1〜3秒のランダムウェイトを設定
- **対策②** User-Agentにプロジェクト名と連絡先メールを明記（誠実なクローラーとして識別させる）
- **対策③** 同一サーバーへの同時アクセス数を制限（Scrapy: `CONCURRENT_REQUESTS_PER_DOMAIN=1`）

### 6.2 JavaScriptレンダリングが必要なサイト

- **対策** Playwright（Pythonバインディング）でヘッドレスブラウザ経由で取得
  - 静的HTMLのみ → requests + BeautifulSoup（高速・軽量）
  - JS必須サイト → Playwright（重いため必要なサイトのみ限定使用）

### 6.3 1,700自治体URLリスト作成

- **Step1** chiholog・地方議会会議録検索サイト（東日本・西日本編）からURL一覧を自動収集
- **Step2** 国立国会図書館リサーチナビのリストで都道府県・政令市を補完
- **Step3** e-Statの自治体コードリストと照合して未収録の町村を特定

## 8. 法的・倫理的留意事項

> ⚠️ **必ず守ること**
> - `robots.txt` を確認し、Disallowされているパスはクロールしない
> - 利用規約でスクレイピング禁止の記載がある場合は対象外とする
> - データは公開情報のみ収集（ログイン不要のページのみ）
> - サーバーに過大な負荷をかけない（クロール間隔・並列数を制限）
> - データの帰属を明示し、出典として各自治体のURLをリンク表示する
