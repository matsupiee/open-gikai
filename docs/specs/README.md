# docs/specs

**現行の仕様** を記述する場所。既に実装されているシステムについて、エージェントや新しいメンバーが仕様を把握するための一次資料。

## いつ書くか

- 外部からの挙動（API スキーマ、CLI インタフェース、スクレイパーの入出力仕様）
- ドメインモデル（議会・議員・議題などの定義と関係）
- 非自明な挙動や制約（レート制限、タイムゾーン、サマリ生成の前提）

## ファイル名

`<領域>-<サブ領域>.md`。例: `api-meetings.md`, `scraping-pdf-strategy.md`。

## 書き方の指針

- 現行の実装に対応する仕様だけを書く。過去の経緯は ADR へ
- コードが一次ソースなので、重複させず **コードでは表現しにくいこと** を書く（why、制約、全体像）
- 仕様が変わったら必ず更新する。ドキュメントが嘘になるとエージェントが騙される
- 古くなったら `docs/deprecated-docs/` に移す
