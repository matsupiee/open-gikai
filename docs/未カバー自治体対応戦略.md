## 対応フェーズ

### Phase 1: 検出ロジック拡張（工数: 小）

既存スクレイパーがそのまま使える可能性が高いもの。`detectSystemType()` にパターンを追加するだけ。

| 対応内容 | 自治体 | 人口 |
|----------|--------|-----:|
| `g08v_search.asp` パターンを gijiroku_com として検出 | 大田区 | 708,478 |
| `db-search.com` を dbsearch として検出 | 小城市・板野町 | 56,264 |

- 大田区は voiweb.exe ベースの ASP で gijiroku.com と同一エンジンの可能性が高い
- `db-search.com` は `dbsr.jp` と同じ大和速記情報センター
- 実際にスクレイパーが動くか検証は必要だが、コード変更は数行

### Phase 2: 新スクレイパー開発（工数: 中）

同一システムでまとめてカバーできるもの。

#### AmiVoice（3 件・人口合計 292,620）

| 自治体 | 人口 | URL |
|--------|-----:|-----|
| 沼津市 | 180,066 | `http://ami-search.amivoice.com/numazu/usr/search.exe?process=detail` |
| 取手市 | 102,938 | `http://ami-search.amivoice.com/toride/usr/search.exe?process=detail` |
| 箱根町 | 9,616 | `http://ami-search.amivoice.com/hakone/usr/search.exe?process=detail` |

- URL 構造が統一されており、スクレイパー 1 本で 3 自治体カバー可能
- テナント名がパスに含まれるため、discussnet_ssp と同様のアーキテクチャで対応できる

## 推奨する進め方

1. **Phase 1** から着手（大田区だけで 70 万人カバー、コード変更は数行）
2. **Phase 3** の URL 調査を並行して進め、既知システムに該当する自治体を発掘
3. **Phase 2** の AmiVoice スクレイパーを開発（3 自治体まとめてカバー）
4. **Phase 4** は人口順に、費用対効果を見ながら個別対応
